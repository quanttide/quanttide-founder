# 工作领域分类规则

## 背景

根据 archive/journal 子模块的文件夹结构，识别作者（创始人）的主要工作领域，建立分类规则。

---

## 一级领域（核心领域）

基于活动类型的一级分类，反映工作的本质属性。

### Think（思考）

**定义**：认知活动，问题的分析、综合、评估。

**子域**：
- `default`：通用思考
- `qtadmin`：管理后台相关思考
- `thera`：数字资产治理工具相关思考

**特征**：产出判断、决策、方案。

**示例**：
- `think/2026-03-20.md`：关于版本升级的反思
- `think/default/qtadmin.md`：管理后台定位思考

---

### Execute（执行）

**定义**：将决策转化为行动的活动。

**子域**：
- `asset`：资产管理执行
- `innov`：创新执行
- `qtclass`：量化课堂执行
- `qtdata`：量化数据执行
- `stdn`：标准化执行

**特征**：产出成果、交付物、进度。

**示例**：
- `execute/qtdata.md`：量化数据平台开发执行

---

### Code（编码）

**定义**：软件开发活动。

**子域**：
- `default`：通用代码
- `iam`：身份认证
- `product`：产品代码
- `stdn`：标准代码

**特征**：产出代码、文档、测试。

**示例**：
- `code/thera.py`：Thera 工具代码
- `code/stdn/`：标准化代码库

---

### Learn（学习）

**定义**：获取知识和技能的活动。

**子域**：
- `default`：通用学习
- `code`：编程学习
- `innov`：创新学习

**特征**：笔记、总结、知识沉淀。

**示例**：
- `learn/code.md`：编程学习笔记

---

### Manage（管理）

**定义**：组织和协调资源的活动。

**子域**：
- `hr`：人力资源管理
- `stdn`：标准化管理
- `write`：写作管理

**特征**：计划、协调、评估。

---

## 二级领域（支撑领域）

支持核心领域的专业领域。

### Product（产品）

**定义**：产品规划和演进。

**子域**：
- `default`：产品通用
- `qtadmin`：管理后台产品

**关系**：被 Think 指引，被 Execute 实现。

---

### Data（数据）

**定义**：数据相关的所有活动。

**子域**：
- `default`：数据通用

**关系**：为所有领域提供数据支撑。

---

### Security（安全）

**定义**：安全相关的所有活动。

**关系**：贯穿所有领域。

---

### Asset（资产）

**定义**：资产管理。

**子域**：
- `founder`：创始人资产
- `github`：GitHub 资产

**关系**：被 Think 规划，被 Execute 管理。

---

## 分类规则

### 规则 1：活动类型优先

当一个条目涉及多个领域时，按活动类型分类：
- 思考 → Think
- 行动 → Execute
- 编码 → Code
- 学习 → Learn
- 组织 → Manage

### 规则 2：多维度标注

使用多个目录标注同一内容，反映多重视角：

**示例**：
- `think/product_qtadmin.md`：关于管理后台产品的思考
- `execute/qtdata.md`：量化数据执行
- `code/thera.py`：Thera 工具代码

### 规则 3：默认域 fallback

无法明确分类时，使用 `default`：

**示例**：
- `think/default.md`：通用思考
- `learn/default.md`：通用学习

### 规则 4：子域继承

子域继承父域的基本属性：

**示例**：
- `think/qtadmin/` 下的文件都是关于管理后台的思考
- `code/stdn/` 下的代码都是标准化的

### 规则 5：专题域独立

具有明确主题的领域独立分类：

- `security/`：安全专题
- `data/`：数据专题
- `asset/`：资产专题

## 命名约定

### 目录命名

- 全小写
- 使用 `stdn` 而非 `standard`（统一缩写）
- 使用 `iam` 而非 `identity`（统一缩写）
- 使用 `mr` 而非 `multi_agent`（统一缩写）

### 文件命名

- 全小写
- 使用 `-` 分隔
- 示例：`2026-03-20.md`、`qtadmin.md`、`founder_asset.md`

### 缩写对照

| 缩写 | 全称 | 含义 |
|------|------|------|
| `stdn` | Standard | 标准 |
| `iam` | Identity & Access Management | 身份与访问管理 |
| `mr` | Multi-Agent | 多智能体 |
| `hr` | Human Resources | 人力资源 |
| `qt` | QuantTide | 量潮 |
| `qtclass` | QuantTide Class | 量潮课堂 |
| `qtdata` | QuantTide Data | 量潮数据 |

## 更新规则

### 新增领域

1. 识别活动的本质属性（一级）
2. 确定支撑领域（二级）或专项领域（三级）
3. 在对应目录创建文件
4. 更新本规则文档

### 合并领域

1. 确认两个领域活动重叠度 > 70%
2. 保留更通用的名称
3. 将旧领域归档到 archive
4. 更新本规则文档

### 删除领域

1. 确认该领域已无活跃产出
2. 归档所有文件到 archive
3. 删除空目录
4. 更新本规则文档

---

*本规则基于 archive 和 journal 的实际文件夹结构归纳，适用于 quanttide-founder 工作空间。*
