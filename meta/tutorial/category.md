# 范畴论为基础的分类和命名哲学

## 从本体论到范畴论

### 本体论：存在的分类

传统分类学基于**本体论**（Ontology），关注"**什么存在**"。

**核心问题**：这个东西是什么？

**经典方法**：
- 本质主义：根据内在属性分类
- 树状结构：父子关系，层层嵌套
- 边界清晰：每个事物有唯一归属

**局限**：
- 僵化：难以处理跨类别事物
- 孤立：忽略事物间的关系
- 静态：无法反映变化和发展

### 范畴论：关系的映射

现代分类学基于**范畴论**（Category Theory），关注"**如何关联**"。

**核心问题**：这个关系是什么？

**核心概念**：
- **对象**（Object）：范畴中的元素
- **态射**（Morphism）：对象间的映射
- **复合**（Composition）：态射的链接
- **函子**（Functor）：范畴间的映射

**优势**：
- 动态：关注关系而非静态属性
- 灵活：一个对象可属于多个范畴
- 系统：通过态射网络理解整体

## meta 目录的范畴论实践

### 作为范畴的 meta 目录

meta 目录本身是一个**范畴**（Category）：

**对象**：
- 文件：IDENTITY.md、SOUL.md、AGENTS.md、TOOLS.md、USER.md
- 文件夹：bylaw/、handbook/、profile/、journal/、report/、brochure/、tutorial/

**态射**：
- 引用：文件之间的链接
- 优先级：层级之间的依赖
- 维护：人机协作的流程

### 程序型记忆五层：层范畴

程序型记忆五层体系是一个**层范畴**（Layered Category）：

| 层次 | 对象 | 态射 | 法律隐喻 |
|------|------|------|----------|
| L1 | Paper | 论证原理 | 立法原理 |
| L2 | Usecase | 案例映射 | 精选判例 |
| L3 | Handbook | 经验提炼 | 权威汇编 |
| L4 | Specification | 规范约束 | 程序性法律 |
| L5 | Bylaw | 规则制定 | 宪法 |

**态射复合**：
```
Paper (原理) → Usecase (案例) → Handbook (手册) → Specification (标准) → Bylaw (章程)
```

### 九宫格模型：积范畴

九宫格模型是一个**积范畴**（Product Category）：

**范畴 1：时间维度**
- 过去
- 现在
- 未来

**范畴 2：类型维度**
- 事件类
- 语义类
- 自我类

**积范畴：时间 × 类型**

| | 事件类 | 语义类 | 自我类 |
|------|--------|--------|--------|
| **过去** | Archive | Tutorial | History |
| **现在** | Journal | Report | Brochure |
| **未来** | Profile | Essay | Roadmap |

**函子映射**：九宫格到子模块

### 记忆分层模型：序范畴

记忆分层模型是一个**序范畴**（Posetal Category）：

**偏序关系**：
```
bylaw > handbook > profile > journal = report
```

**态射**：
- 更新依赖：高层级约束低层级
- 优先级：高层级优先于低层级
- 继承：低层级遵循高层级规则

## 范畴论命名哲学

### 命名原则

#### 1. 对象命名：本质 vs 关系

**本体论命名**：
```
file1.md, file2.md, file3.md
```
- 问题：无语义，无关系

**范畴论命名**：
```
IDENTITY.md, SOUL.md, AGENTS.md
```
- 优势：每个名字标识一个角色和关系

#### 2. 层级命名：嵌套 vs 扁平

**本体论嵌套**：
```
meta/bylaw/report.md
meta/profile/release.md
```
- 树状结构，单一归属

**范畴论扁平**：
```
bylaw.md (最高优先级)
report.md (规范)
release.md (状态)
```
- 网状结构，多重关联

#### 3. 态射命名：引用 vs 语义

**本体论引用**：
```
See also: file.md
```

**范畴论语义**：
```
详见 [记忆建模](./profile/memory.md)
```
- 明确态射类型：引用
- 明确态射目标：profile/memory.md

### 实践案例

#### 案例 1：IDENTITY.md 的命名

**本体论视角**：
- 它是"元数据文档"
- 描述"仓库状态"

**范畴论视角**：
- 它是"自我映射"对象
- 态射到：
  - profile/memory.md（引用记忆建模）
  - profile/release.md（引用版本信息）
  - submodule.md（引用子模块列表）

**哲学意义**：
- 名字不只是描述，而是关系节点
- "IDENTITY" = 自我映射 = 与所有元数据的关系

#### 案例 2：bylaw/ 的命名

**本体论视角**：
- 它是"规则文档"
- 描述"工作章程"

**范畴论视角**：
- 它是"约束"对象
- 态射：
  - 规范所有低层文档（优先态射）
  - 被人类制定（制定态射）
  - 被 handbook/ 继承（继承态射）

**哲学意义**：
- 名字反映在范畴中的角色
- "bylaw" = 约束 = 范畴的态射源

#### 案例 3：程序型记忆五层的命名

**本体论视角**：
- 五个不同的文档类型
- 从论文到章程的等级

**范畴论视角**：
- 五个层次的态射序列
- 每层是前一层的"提升"态射
- 整体形成一个"提升"函子

**哲学意义**：
- 命名反映态射链
- 层次 = 态射 = 认知深化

## 范畴论分类优势

### 1. 动态适应

**本体论**：
- 新事物 → 重新分类树
- 代价：高

**范畴论**：
- 新事物 → 添加新态射
- 代价：低

**案例**：
- 新增 `tutorial/` 目录
- 本体论：需要重新设计目录树
- 范畴论：只需添加从 brochure 的态射

### 2. 多重归属

**本体论**：
- 每个事物有唯一归属
- 问题：跨类别事物难处理

**范畴论**：
- 一个对象可有多个态射
- 优势：灵活表达复杂关系

**案例**：
- `AGENTS.md` 既是导航文件，也是自我认知文档
- 态射到：根目录 AGENTS.md（引用）、meta/SOUL.md（认知）、meta/TOOLS.md（工具）

### 3. 系统理解

**本体论**：
- 通过分类理解个体
- 问题：丢失整体关系

**范畴论**：
- 通过态射网络理解系统
- 优势：把握整体结构

**案例**：
- 通过态射图理解 meta 目录：
  - bylaw → handbook → profile → journal
  - IDENTITY → profile/*（引用态射）
  - AGENTS → SOUL（认知态射）

## 应用指南

### 如何设计范畴论分类

#### 步骤 1：识别对象

**问题**：系统中存在什么？

**实践**：
- 列出所有文件和文件夹
- 识别每个对象的角色

**meta 案例**：
- 文件：IDENTITY、SOUL、AGENTS、TOOLS、USER
- 文件夹：bylaw、handbook、profile、journal、report

#### 步骤 2：定义态射

**问题**：对象间如何关联？

**实践**：
- 识别引用关系
- 识别依赖关系
- 识别优先级关系

**meta 案例**：
- 引用：IDENTITY → profile/memory.md
- 依赖：bylaw → handbook → profile
- 优先级：bylaw > handbook > profile

#### 步骤 3：构建范畴

**问题**：如何组织对象和态射？

**实践**：
- 识别子范畴（如时间维度、类型维度）
- 识别积范畴（如九宫格）
- 识别层范畴（如五层体系）

**meta 案例**：
- 子范畴：时间维度、类型维度
- 积范畴：时间 × 类型 = 九宫格
- 层范畴：程序型记忆五层

#### 步骤 4：命名对象

**问题**：如何命名反映关系？

**实践**：
- 名字反映在范畴中的角色
- 名字反映与其他对象的关系
- 名字反映态射链中的位置

**meta 案例**：
- IDENTITY：自我映射对象
- bylaw：约束对象
- profile：状态对象

### 常见错误

#### 错误 1：过度分类

**本体论思维**：
```
meta/
├── rules/
│   ├── bylaw/
│   ├── handbook/
│   └── ...
├── metadata/
│   ├── identity/
│   └── ...
```

**范畴论思维**：
```
meta/
├── bylaw/      # 直接命名，反映角色
├── handbook/
├── IDENTITY.md # 直接命名，反映关系
```

#### 错误 2：忽略态射

**本体论思维**：
```
file1.md
file2.md
file3.md
```

**范畴论思维**：
```
IDENTITY.md  # 引用态射
SOUL.md      # 认知态射
AGENTS.md    # 协作态射
```

#### 错误 3：静态分类

**本体论思维**：
- 目录结构固定
- 难以适应变化

**范畴论思维**：
- 态射网络灵活
- 易于添加新关系

## 总结

### 本体论 vs 范畴论

| 维度 | 本体论 | 范畴论 |
|------|--------|--------|
| 关注点 | 什么存在 | 如何关联 |
| 方法 | 属性分类 | 态射映射 |
| 结构 | 树状 | 网络 |
| 优势 | 清晰简单 | 灵活系统 |
| 局限 | 僵化孤立 | 复杂抽象 |

### meta 目录的启示

**meta 目录展示了范畴论分类的实践**：

1. **对象命名反映关系**：IDENTITY、SOUL、AGENTS
2. **层级反映态射**：bylaw > handbook > profile
3. **维度反映积范畴**：时间 × 类型 = 九宫格
4. **灵活反映动态**：易添加新对象和态射

### 哲学意义

**从本体论到范畴论**：
- 从"存在"到"关系"
- 从"静态"到"动态"
- 从"分类"到"映射"

**meta 目录的命名哲学**：
- 名字不只是标签，而是关系节点
- 分类不只是分组，而是态射网络
- 结构不只是层次，而是认知映射
