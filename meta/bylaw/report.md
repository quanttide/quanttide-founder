# 元报告章程

本文件定义 quanttide-founder 仓库中日报/总结（report）的结构规范。

## 区分说明

- **原始事件记录**（`meta/journal/`）：按时间顺序记录的实际发生事件
- **日报/总结**（`meta/report/`）：按记忆分类提炼的总结性报告

## 报告结构

### 一级标题：日期

日志文件以日期作为一级标题，格式为 `YYYY-MM-DD`。

```
# 2026-03-17
```

### 二级标题：记忆分类

日志内容按记忆分类框架组织，包含三个二级标题：

1. **事件记忆**：记录发生的具体事件
2. **语义记忆**：从事件中提炼的知识和规律
3. **自我记忆**：基于事件和语义的自我认知和反思

### 三级标题及以下：事件细节

在"事件记忆"部分，根据具体事件类型使用三级标题分类：

#### 工作报告（report）

- **执行任务**：记录具体执行的任务
- **遇到问题**：记录遇到的问题和挑战
- **解决方案**：记录问题的解决方案
- **成果输出**：记录工作成果

#### 示例结构

```
# 2026-03-17

## 事件记忆

### 执行任务
- 完成元数据重构工作
- 发布主仓库 v0.1.0 Release

### 遇到问题
- CHANGELOG.md 文件被删除
- 需要更新元数据文件

### 解决方案
- 提交 CHANGELOG.md 删除操作
- 创建新的元数据文件并更新引用

### 成果输出
- meta/profile/release.md
- meta/profile/submodule.md
- meta/profile/memory.md
- meta/profile/repo.md
- meta/IDENTITY.md 重构

## 语义记忆

### 知识提炼
- 元数据重构原则：将详细信息移到专门文件，主文件保留导航
- Release 流程：创建标签 → 准备 Notes → 创建 Release → 更新元数据
- 文件职责分离：IDENTITY.md 负责导航，profile/*.md 负责细节

### 规律总结
- 日志结构应遵循记忆分类框架
- 事件记忆 → 语义记忆 → 自我记忆 的提炼顺序
- 版本发布需要完整的流程和文档更新

## 自我记忆

### 认知更新
- 认识到元数据重构的重要性：提高文档可维护性
- 理解 Release 流程的完整性：确保版本管理规范
- 体会职责分离的好处：提高代码可读性和可维护性

### 行为反思
- 应更加注重文档的结构化和规范化
- 在发布版本时，要确保所有相关文档都已更新
- 定期回顾和优化工作流程
```

## 编写规范

### 事件记忆编写原则

1. **客观记录**：只记录发生的事实，不添加主观评价
2. **具体详细**：包含时间、地点、人物、事件等细节
3. **分类清晰**：按任务类型、问题类型等分类组织

### 语义记忆提炼原则

1. **从事件中提炼**：基于具体事件总结规律和知识
2. **抽象概括**：将具体事件上升为一般性原则
3. **分类组织**：按知识类型分类组织

### 自我记忆提炼原则

1. **基于事件和语义**：结合具体事件和提炼的知识
2. **认知更新**：记录新的认识和理解
3. **行为反思**：反思行为模式，制定改进计划
