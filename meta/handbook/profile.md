# Profile 维护手册

本文件记录 quanttide-founder 仓库中 `meta/profile/` 目录的维护经验和规范。

## 维护经验总结

### 可以遗忘的内容

#### 1. 错误的文件创建步骤
- 创建 `meta/profile/default/` 目录（错误的目录结构）
- 临时创建的 `meta/profile/default/semantic_memory.md` 文件
- **原因**：这些是试错过程中的临时产物，已正确修正

#### 2. 冗余的信息收集
- 在 `default.md` 中重复收集已存在文件中的信息
- **原因**：信息应该只存储一次，避免冗余

#### 3. 不合理的分类尝试
- 最初将所有语义记忆集中到一个文件的想法
- **原因**：不符合单一职责原则，不利于维护

### 应该保留的经验

#### 1. 文件职责分离原则
```
repo.md      → 仓库结构 + 元数据管理
release.md   → Release 列表 + 版本管理  
memory.md    → 记忆建模 + 日志管理 + 元数据规范
submodule.md → 子模块列表 + 工作流程
```

**保留理由**：清晰的职责分离提高可维护性

#### 2. 内容分类逻辑
| 内容类型 | 适合存放的文件 |
|---------|---------------|
| 仓库结构规范 | repo.md |
| 版本管理流程 | release.md |
| 记忆建模知识 | memory.md |
| 子模块工作流 | submodule.md |

**保留理由**：合理的分类便于查找和维护

#### 3. 引用机制
- 主文件 `IDENTITY.md` 保留导航引用
- 详细内容分散到 `profile/*.md` 文件

**保留理由**：避免信息冗余，保持主文件简洁

#### 4. 错误修正流程
- 发现错误 → 立即修正 → 提交验证
**保留理由**：快速迭代的工作习惯

## 维护流程规范

### 1. 文件创建检查清单
在创建新文件前，先检查：
- 是否已有类似内容的文件？
- 内容是否符合文件职责？
- 是否需要引用而不是复制？

### 2. 内容分类原则
- **单一职责**：每个文件只负责一个方面的内容
- **逻辑清晰**：分类符合直觉，便于查找
- **避免冗余**：相同信息只存储一次

### 3. 工作流程
```
1. 分析内容性质
2. 选择合适的文件
3. 检查是否需要引用
4. 创建/更新文件
5. 验证分类合理性
```

## 常见错误及避免方法

### 错误 1：创建冗余目录
**现象**：创建 `meta/profile/default/` 目录
**避免**：先检查是否需要目录，还是直接创建文件

### 错误 2：信息重复存储  
**现象**：在多个文件中存储相同信息
**避免**：使用引用机制，避免复制粘贴

### 错误 3：文件职责不清
**现象**：一个文件包含多种不相关的内容
**避免**：遵循单一职责原则，合理分类
